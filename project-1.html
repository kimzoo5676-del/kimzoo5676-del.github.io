배경 & 문제

예·적금을 고르다 보면 은행마다 우대조건 표현이 제각각이고, 약관을 읽어도 결국 내가 얼마를 받는지 알기 어렵습니다. 그 계산과 해석을 대신해 주는 서비스를 만들기로 했습니다. 2인 팀에서 백엔드와 AI를 전담해, 데이터 수집부터 추천 로직, LLM 연동, 배포까지 맡았습니다.

한 일

  • Django REST Framework로 REST API와 JWT 인증을 구현하고, 회원·상품·가입·찜·커뮤니티 도메인의 DB(ERD)를 모델링했습니다.
  • 금융감독원(FSS) API에서 예·적금 상품 766건을 수집하고, 조회 프로필 기반 필터링 + 세후 실수령액 계산·정렬로 추천 알고리즘을 설계·구현했습니다.
  • LLM(GPT-5-mini)을 세 가지로 연동했습니다 — ① 은행별 자유 텍스트 우대조건을 8개 표준 태그(JSON)로 배치 구조화(요청마다 LLM을 부르면 응답이 수십 초로 느려져, 미리 가공해 추천은 DB 쿼리만으로 즉시 응답), ② 약관을 쉬운 말로 요약, ③ 해당 상품 데이터로 근거를 한정한 실시간 스트리밍 챗봇.
  • GitHub Actions로 CI/CD를 구성하고 AWS EC2(Nginx·Gunicorn)에 배포한 뒤 실제 도메인을 연결해, 로컬에서 프로덕션까지 서비스 생명주기 전체를 경험했습니다.

결과 & 배운 점

삼성청년SW·AI아카데미 1학기 관통 프로젝트 최우수상(서울2반 1등)을 받았고, 조건 입력 한 번으로 766개 상품을 세후 실수령액 기준으로 정렬해 주는 서비스를 실제 도메인에 배포했습니다. 가장 크게 배운 건 우대조건 구조화에서였습니다. 처음엔 정규식으로 뽑으려 했지만, "최대 연 5.0% 우대금리가 적용돼요" 같은 홍보 문구가 조건으로 잡히는 식의 오류가 은행마다 다른 표현만큼 쏟아졌습니다. LLM 구조화로 바꾼 뒤에도 API 예산의 절반을 쓴 시점에 전부 뽑겠다는 욕심을 버리고, 실제 데이터를 하나씩 확인하며 가장 보편적인 조건 8개만 표준 태그로 좁혔습니다. 가정하지 말고 데이터를 확인한다는 원칙은 이때 몸으로 배웠습니다.
